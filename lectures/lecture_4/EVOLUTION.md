# **Эволюция RAG-системы**

## Цель работы

Собрать минимально корректный baseline RAG для ответов по двум документам с потребительскими настроениями за 2019 и 2025 годы.

При реализации я сознательно не делал сложную multi-agent архитектуру, reranker и другие тяжелые улучшения на первом шаге. Сначала была задача собрать рабочий vanilla RAG, который:

- готовит документы для индексации
- разбивает текст на чанки
- векторизует чанки
- загружает их в LanceDB
- извлекает релевантные фрагменты по вопросу
- генерирует ответ только на основе найденного контекста
- возвращает ответ в формате, совместимом с `evaluation.py`

**Ограничения**: суммарный контекст, передаваемый в генерацию, не должен превышать 6000 символов.

# Baseline MVP

## Шаг 0 - Базовый baseline

### Что было реализовано
Собран базовый RAG-пайплайн из следующих частей:

1. `data_preporation.py`
   - чтение PDF-документов из папки `dataset`
   - извлечение текста
   - базовая очистка текста
   - извлечение года документа из имени файла
   - передача текста в модуль чанкинга

2. `chunking.py`
   - для baseline выбрана стратегия `semantic_chunking`
   - текст разбивается через `RecursiveCharacterTextSplitter`
   - в каждый чанк прокидываются метаданные документа, включая `year`

3. `embedder.py`
   - один embedder используется и для документов, и для запроса пользователя

4. `lance_db.py`
   - создается таблица `chunks` в `lance_db/vectorstore`
   - в таблицу сохраняются текст чанка, embedding и метаданные

5. `retriever.py`
   - вопрос преобразуется в embedding
   - выполняется semantic search по векторной базе
   - результаты дедуплицируются
   - итоговый контекст обрезается по лимиту 6000 символов

6. `agent.py`
   - реализован тонкий слой над retrieval + generation
   - агент возвращает `RAGAgentAnswer`
   - retrieved chunks отдаются в формате, который понимает модуль оценки

### Почему именно такой baseline
На первом шаге важно было получить устойчиво работающую систему, совместимую с ноутбуком проверки и модулем `evaluation.py`.

Этот baseline решает сразу несколько задач:
- соответствует структуре домашнего задания
- не ломает существующий проект
- проходит через ожидаемый интерфейс `RAGAgent`
- поддерживает `retrieved_chunks`, которые нужны для retrieval-метрик и faithfulness

## Шаг 1 - Выбор семантического чанкинга

### Описание

Для baseline использован `semantic_chunking`, а не фиксированная нарезка по символам.

В исходных PDF данные организованы как аналитический текст с абзацами, заголовками и пояснениями к графикам. Если резать документ слишком грубо, то фраза с ответом может разрываться посередине, а retrieval начинает вытаскивать неполные куски.

`semantic_chunking` лучше подходит для такого корпуса, потому что сначала пытается резать по:
- пустым строкам
- строкам
- окончаниям предложений
- пробелам

### Что изменилось

- использован `RecursiveCharacterTextSplitter`
- сохранены метаданные документа для каждого чанка
- baseline строится на одном понятном методе, без смешения нескольких стратегий чанкинга

**Метрики** - Отдельный замер "до fixed-size / после semantic" не сохранялся, поэтому численную дельту по этому шагу я не фиксирую.

**Вывод** - Для первого baseline семантический чанкинг оказался разумным выбором: он лучше сохраняет смысл внутри чанка и не усложняет систему.

## Шаг 2 - Протаскивание метаданных документа

### Описание
Во все чанки был добавлен `year`, соответствующий исходному документу.

Тестовый датасет содержит колонку `document`, где зашит год отчета: 2019 или 2025. Модуль `evaluation.py` использует это значение как `expected_year` и сравнивает его с `chunk.year`.

Если retrieval находит текст из не того года, то даже семантически похожий фрагмент может считаться нерелевантным.

### Что изменилось
- год извлекается из имени PDF
- год сохраняется в metadata чанка
- год попадает в векторную таблицу
- год возвращается наружу в `retrieved_chunks`

**Метрики** - Отдельный промежуточный замер по этому шагу не сохранялся.

**Вывод** - Это необходимое условие для корректной оценки. Без этого шага retrieval-метрики были бы искажены.

## Шаг 3 - Ограничение контекста до 6000 символов

### Описание

В retriever добавлено ограничение на суммарную длину контекста: в генерацию передаются только те top-k чанки, которые помещаются в лимит 6000 символов.

Это прямое ограничение из дз. Кроме того, слишком длинный контекст ухудшает интерпретируемость baseline: модель может отвечать за счет большого объема лишнего текста, и тогда качество retrieval замеряется менее честно.

### Что изменилось
- retrieval сначала получает top-k кандидатов
- затем чанки добавляются в контекст последовательно
- как только лимит по символам превышается, добавление прекращается

**Метрики**
Отдельного замера "без лимита / с лимитом" не проводилось.

**Вывод** 
Этот шаг был нужен для соблюдения условий задания и более корректной постановки эксперимента.

- retrieval
- сбор контекста
- один вызов LLM на генерацию ответа

## Уточнения по контракту evaluator
 
После более внимательного чтения `evaluation.py` зафиксированы два важных факта, которые определяют дизайн всех следующих гипотез:
 
### Факт 1. Метрики считаются по всему `retrieved_chunks`, не по фиксированному top-K
 
`evaluate()` вызывает `_precision_at_k(rel_vec)` и `_recall_at_k(...)` без явного `k`. То есть `Precision@K` в названии — это precision по всей длине `retrieved_chunks`, которые агент вернул в `RAGAgentAnswer`.
 
**Следствие:** каждый лишний чанк, попавший в `retrieved_chunks`, напрямую штрафует Precision. Задача retriever — вернуть **минимально достаточный** список, а не максимально полный.
 
### Факт 2. Релевантность считается через год и token-overlap
 
`_get_relevance_vector()` сначала отбрасывает чанки с `chunk.year != expected_year` (сразу 0), затем использует greedy one-to-one matching: каждый ground-truth чанк "тратится" на первый retrieved, у которого `_token_overlap_recall(chunk.text, gt) >= 0.3`.
 
**Следствие 1:** любой чанк не из того года = гарантированный 0 в Precision. Year-filter — самый дешёвый и сильный буст.
 
**Следствие 2:** дубликаты и почти-дубликаты бесполезны. Второй похожий чанк не получит балл за тот же gt, потому что gt уже "потрачен", но при этом он добавит 0 в relevance vector и опустит Precision.
 
### Факт 3. `retrieved_chunks` и контекст для LLM должны быть одним списком
 
Faithfulness считается по `ans.retrieved_chunks`, а не по тому, что реально увидела модель. Если урезать только контекст для LLM, а наружу вернуть больше — получится рассинхрон между тем, на чём генерировался ответ, и тем, по чему его оценивают.
 
**Следствие:** retriever должен формировать **один финальный список**, который одновременно идёт и в LLM, и в `RAGAgentAnswer.retrieved_chunks`.

## Следующие шаги развития
 
План разбит на три независимых эксперимента (A, B, C), которые делаются и замеряются по очереди. Это позволяет видеть чистый эффект каждой правки и быстро откатывать ту, что ломает Recall.
 
### Эксперимент A — Year-aware retrieval
 
**Проблема.** Согласно `_get_relevance_vector`, чанк из неправильного года получает 0 в relevance vector без проверки текста. Сейчас retriever тащит кандидатов из обоих годов вперемешку, и часть из них механически штрафует Precision.
 
**Идея.**
- Парсить из вопроса год регуляркой по `2019|2025` (других годов в корпусе нет).
- Если год найден — добавлять `.where("year = N")` к запросу LanceDB до `.limit()`.
- **Fallback:** если фильтр вернул 0 строк, повторять запрос без фильтра, чтобы не возвращать пустой ответ на вопросы без явного года или с опечаткой.
 
**Ожидаемый эффект.** Сильный рост Precision@K и F1@K. Recall почти не меняется (релевантные чанки и так из правильного года).
 
**Stop-rule.** Если Recall падает больше чем на 0.03 — проверить, что fallback на пустой результат действительно срабатывает.
 
---
 
### Эксперимент B — Перепаковка retriever: один финальный список
 
**Проблема.** Сейчас в `retriever.retrieve` логика обрыва по бюджету неоптимальна (`break` вместо `continue`), бюджет считается по сырому тексту без учёта обёртки `[Фрагмент N | год YYYY]`, и в `agent._build_context` контекст формируется отдельно от того, что уходит в `retrieved_chunks`. Это даёт сразу три проблемы: возможное превышение лимита 6000, потеря коротких чанков после длинного, и потенциальный рассинхрон между LLM и evaluator.
 
**Идея.** Сделать retriever единственной точкой формирования финального списка чанков.
- Запрашивать из БД больше кандидатов: `top_k = 20-25`.
- Делать exact-dedup по тексту (как сейчас).
- Делать budget packing с `continue` вместо `break`: если текущий чанк не влез — пробовать следующий, более короткий.
- Считать бюджет 6000 символов **по реальному финальному контексту с обёрткой**, а не по сырому тексту.
- Возвращать ровно один список, который потом и идёт в `_build_context`, и кладётся в `RAGAgentAnswer.retrieved_chunks` без изменений.
 
**Ожидаемый эффект.** Precision вверх (нет лишних чанков сверх бюджета). Faithfulness может слегка вырасти (LLM и evaluator видят одно и то же). Recall сохраняется или слегка растёт за счёт более умной упаковки.
 
**Stop-rule.** Если Faithfulness падает — проверить, что список действительно один и тот же в обоих местах.
 
---
 
### Эксперимент C — Near-duplicate dedup по соседним `chunk_index`
 
**Проблема.** Из-за overlap=150 соседние чанки одного документа сильно пересекаются по тексту. В greedy matching evaluator-а только один из них получит балл за общий gt-чанк, а второй добавит 0 в relevance vector и опустит Precision.
 
**Идея.** При дедупликации в retriever дополнительно отбрасывать чанк, если в уже отобранных есть чанк с тем же `document_name` и `|chunk_index - existing_chunk_index| <= 1`.
 
**Ожидаемый эффект.** Precision вверх. Recall может немного просесть, потому что иногда соседний чанк действительно нужен.
 
**Stop-rule.** Это самая агрессивная правка из трёх. Если Recall падает больше чем на 0.05 — откатывать первой. Запускать только после успешных A и B.
 
---
 
### Что отложено за рамками текущего бюджета
 
Эти улучшения остаются разумными, но требуют новых зависимостей или существенно больше кода. Откладываю их до того момента, как простые правки A/B/C исчерпают себя.
 
- **Reranker поверх top-k.** Классический ход для recall-oriented retriever-а. Подняли бы Precision, MAP, NDCG. Минусы: новая зависимость, латентность, новый код. Делать только если после A/B/C метрики всё ещё неудовлетворительны.
- **Hybrid retrieval (BM25 + dense).** Помог бы на вопросах с точными формулировками и числами. Минус: LanceDB <0.20 имеет нестабильный FTS API, риск сломать ingest.
- **Изменение размера чанка.** Соблазнительно после понимания, что релевантность считается через token-overlap. Но требует пересборки всей БД и риска сломать текущий Recall 0.94. Не трогать, пока не выжаты A/B/C.
 
---
 
### Общий stop-rule по серии экспериментов
 
После каждого шага замеряется полный набор метрик: Precision, Recall, F1, MRR, MAP, NDCG, Faithfulness.
 
Критерий:
- если Precision растёт, а Recall падает не больше чем на 0.03–0.05 — правка принимается
- если Recall падает сильнее — правка откатывается, начиная с эксперимента C
- если Faithfulness падает — в первую очередь проверяется синхрон между контекстом LLM и `retrieved_chunks` (эксперимент B)


# Эксперимент A: year-aware retrieval

### Что было сделано

В `retriever.py` добавлен year-aware поиск:

- регулярка `\b(2019|2025)\b` извлекает год из вопроса
- если год найден — к запросу LanceDB добавляется `.where("year = N")` до `.limit()`
- если фильтр вернул 0 строк или упал с исключением — fallback на обычный поиск без фильтра
- если года в вопросе нет — обычный поиск, как раньше

Никакие другие части пайплайна не трогались. Это чистый замер эффекта одной правки.

### Метрики до и после

| Метрика      | Baseline | Эксперимент A | Δ        |
|--------------|----------|----------------|----------|
| Precision@K  | 0.1356   | 0.1419         | +0.006   |
| Recall@K     | 0.9444   | 0.9800         | +0.036   |
| F1@K         | 0.2350   | 0.2450         | +0.010   |
| MRR          | 0.7024   | 0.8129         | **+0.110** |
| MAP          | 0.6829   | 0.7957         | **+0.113** |
| NDCG@K       | 0.7528   | 0.8464         | **+0.094** |
| Faithfulness | 0.7463   | 0.8868         | **+0.140** |

### Что неожиданно

Изначально я ждал основной буст в Precision и F1. По факту получилось наоборот: Precision и F1 почти не сдвинулись, а сильнее всего выросли MRR, MAP, NDCG и Faithfulness.

### Интерпретация

**1. Precision не вырос, потому что он упирается не в шум, а в количество возвращаемых чанков.**

В distribution метрик хорошо видно:
- `max precision = 0.4286` ≈ 3/7
- `median = 0.1429` ≈ 1/7
- худшие вопросы стабильно дают precision = 0.1111 ≈ 1/9

То есть retriever почти всегда возвращает 7-9 чанков, а релевантным в них оказывается обычно один. Year-filter заменил "чанки из неправильного года" на "чанки из правильного года, но не те, что нужны". Знаменатель Precision не изменился.

**2. MRR / MAP / NDCG выросли, потому что релевантный чанк всплыл наверх.**

Эти метрики чувствительны к позиции релевантного чанка в выдаче. После year-filter релевантный чанк перестали "теснить" семантически близкие куски из неправильного года, и он стал стабильно появляться на первых позициях. Это сильный сигнал, что ранжирование теперь работает корректно.

**3. Faithfulness вырос на +0.14 — самый крупный прирост в эксперименте.**

Раньше LLM получала смесь "в 2019 было X" + "в 2025 было Y" и путалась на стыках, добавляя факты, которых в нужном году не было. После year-filter контекст стал когерентным по году, и модель перестала склеивать данные между периодами.

**4. Худшие вопросы — это вопросы с явно заданным 2025.**

Все 5 худших вопросов по F1 — формулировки вида "Согласно данным 2025 года...", "В 2025 году какая группа...". У них Recall = 1.0 и Faithfulness = 1.0, но Precision = 0.1111. Это значит, что year-filter на них сработал идеально, но они всё равно возвращают 9 чанков, из которых релевантен один. Это прямое подтверждение, что bottleneck — в размере итогового списка, а не в качестве фильтрации.

### Вывод по эксперименту A

Эксперимент засчитан как успешный, но диагноз бутылочного горлышка обновился:
- retrieval теперь корректно ранжирует и фильтрует по году
- генерация перестала галлюцинировать на стыках годов
- но Precision и F1 упираются в размер `retrieved_chunks`, а не в шум внутри него

Это меняет постановку Эксперимента B.

### Как это меняет Эксперимент B

Изначально B формулировался как "правильная упаковка бюджета 6000 символов и синхрон контекста LLM с retrieved_chunks". Эта часть остаётся в силе, но главным становится другое: **жёсткое ограничение на размер итогового списка чанков**.

Логика простая. Recall = 0.98 и MRR = 0.81 говорят, что нужный чанк почти всегда уже на первых 1-3 позициях. Значит, если вернуть не 7-9 чанков, а 5, Precision механически вырастет с ~1/7 до ~1/5, а Recall просядет минимально.

**Уточнённый план Эксперимента B:**

1. Запрашивать из БД `top_k = 20` с year-фильтром (как сейчас)
2. Делать exact dedup по тексту
3. **Жёстко обрезать итоговый список до первых 5 чанков по distance**
4. Бюджет 6000 символов оставить как safety net (при cap=5 и средней длине ~900 он почти не будет срабатывать)
5. Этот же список идёт и в LLM, и в `RAGAgentAnswer.retrieved_chunks` — никакого рассинхрона

**Ожидание:**
- Precision: 0.14 → ~0.25-0.30
- Recall: 0.98 → ~0.92-0.95 (допустимая просадка)
- F1: 0.25 → ~0.35-0.40
- MRR / MAP / NDCG: сохранятся или слегка вырастут
- Faithfulness: сохранится около 0.88

**Stop-rule:** если Recall падает ниже 0.90, поднять cap с 5 до 6 или 7 и перезамерить.
