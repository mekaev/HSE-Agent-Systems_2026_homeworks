# Эволюция RAG-системы

> Тут описываете как развивали ваш RAG, какие методы улучшения решили применить, почему и что это дало. Если ничего не дало то тоже пишите

## Цель работы

Собрать минимально корректный baseline RAG для ответов по двум документам с потребительскими настроениями за 2019 и 2025 годы.

При реализации я сознательно не делал сложную multi-agent архитектуру, reranker и другие тяжелые улучшения на первом шаге. Сначала была задача собрать рабочий vanilla RAG, который:

- готовит документы для индексации
- разбивает текст на чанки
- векторизует чанки
- загружает их в LanceDB
- извлекает релевантные фрагменты по вопросу
- генерирует ответ только на основе найденного контекста
- возвращает ответ в формате, совместимом с `evaluation.py`

**Ограничения**: суммарный контекст, передаваемый в генерацию, не должен превышать 6000 символов.

---

## Шаг 0 - Базовый baseline

### Что было реализовано
Собран базовый RAG-пайплайн из следующих частей:

1. `data_preporation.py`
   - чтение PDF-документов из папки `dataset`
   - извлечение текста
   - базовая очистка текста
   - извлечение года документа из имени файла
   - передача текста в модуль чанкинга

2. `chunking.py`
   - для baseline выбрана стратегия `semantic_chunking`
   - текст разбивается через `RecursiveCharacterTextSplitter`
   - в каждый чанк прокидываются метаданные документа, включая `year`

3. `embedder.py`
   - один embedder используется и для документов, и для запроса пользователя

4. `lance_db.py`
   - создается таблица `chunks` в `lance_db/vectorstore`
   - в таблицу сохраняются текст чанка, embedding и метаданные

5. `retriever.py`
   - вопрос преобразуется в embedding
   - выполняется semantic search по векторной базе
   - результаты дедуплицируются
   - итоговый контекст обрезается по лимиту 6000 символов

6. `agent.py`
   - реализован тонкий слой над retrieval + generation
   - агент возвращает `RAGAgentAnswer`
   - retrieved chunks отдаются в формате, который понимает модуль оценки

### Почему именно такой baseline
На первом шаге важно было получить устойчиво работающую систему, совместимую с ноутбуком проверки и модулем `evaluation.py`.

Этот baseline решает сразу несколько задач:
- соответствует структуре домашнего задания
- не ломает существующий проект
- проходит через ожидаемый интерфейс `RAGAgent`
- поддерживает `retrieved_chunks`, которые нужны для retrieval-метрик и faithfulness

## Шаг 1 - Выбор семантического чанкинга

### Описание

Для baseline использован `semantic_chunking`, а не фиксированная нарезка по символам.

В исходных PDF данные организованы как аналитический текст с абзацами, заголовками и пояснениями к графикам. Если резать документ слишком грубо, то фраза с ответом может разрываться посередине, а retrieval начинает вытаскивать неполные куски.

`semantic_chunking` лучше подходит для такого корпуса, потому что сначала пытается резать по:
- пустым строкам
- строкам
- окончаниям предложений
- пробелам

### Что изменилось

- использован `RecursiveCharacterTextSplitter`
- сохранены метаданные документа для каждого чанка
- baseline строится на одном понятном методе, без смешения нескольких стратегий чанкинга

**Метрики** - Отдельный замер "до fixed-size / после semantic" не сохранялся, поэтому численную дельту по этому шагу я не фиксирую.

**Вывод** - Для первого baseline семантический чанкинг оказался разумным выбором: он лучше сохраняет смысл внутри чанка и не усложняет систему.

## Шаг 2 - Протаскивание метаданных документа

### Описание
Во все чанки был добавлен `year`, соответствующий исходному документу.

Тестовый датасет содержит колонку `document`, где зашит год отчета: 2019 или 2025. Модуль `evaluation.py` использует это значение как `expected_year` и сравнивает его с `chunk.year`.

Если retrieval находит текст из не того года, то даже семантически похожий фрагмент может считаться нерелевантным.

### Что изменилось
- год извлекается из имени PDF
- год сохраняется в metadata чанка
- год попадает в векторную таблицу
- год возвращается наружу в `retrieved_chunks`

**Метрики** - Отдельный промежуточный замер по этому шагу не сохранялся.

**Вывод** - Это необходимое условие для корректной оценки. Без этого шага retrieval-метрики были бы искажены.

## Шаг 3 - Ограничение контекста до 6000 символов

### Описание

В retriever добавлено ограничение на суммарную длину контекста: в генерацию передаются только те top-k чанки, которые помещаются в лимит 6000 символов.

Это прямое ограничение из дз. Кроме того, слишком длинный контекст ухудшает интерпретируемость baseline: модель может отвечать за счет большого объема лишнего текста, и тогда качество retrieval замеряется менее честно.

### Что изменилось
- retrieval сначала получает top-k кандидатов
- затем чанки добавляются в контекст последовательно
- как только лимит по символам превышается, добавление прекращается

**Метрики**
Отдельного замера "без лимита / с лимитом" не проводилось.

**Вывод** 
Этот шаг был нужен для соблюдения условий задания и более корректной постановки эксперимента.

## Шаг 4 - Совместимость с ноутбуком проверки

### Описание
Несмотря на то что внутри реализован обычный vanilla RAG, внешний интерфейс был оформлен так, как ожидает `evaluation_example.ipynb`.

### Что было сделано
- сохранены `RAGAgent`, `RAGAgentAnswer`
- сохранены `TOOLS_SCHEMA` и `get_tool_functions(...)`
- реализован `run(question, verbose=False, dataset_row_id=...)`

### Обоснование

Ноутбук проверки импортирует именно эти сущности. Поэтому пришлось сделать тонкую tool-обертку вокруг retrieval. При этом внутри логика осталась простой:
- retrieval
- сбор контекста
- один вызов LLM на генерацию ответа

## Итоговые метрики baseline

По итоговому прогону на тестовом датасете были получены следующие средние значения:

| Метрика      | Значение |
|--------------|----------|
| Precision@K  | 0.1356   |
| Recall@K     | 0.9444   |
| F1@K         | 0.2350   |
| MRR          | 0.7024   |
| MAP          | 0.6829   |
| NDCG@K       | 0.7528   |
| Faithfulness | 0.7463   |

---

## Интерпретация результатов

### 1. Очень высокий Recall@K = 0.9444
Это главный плюс baseline.

Система почти всегда находит нужный фрагмент среди retrieved chunks. Значит:
- документы действительно корректно проиндексированы
- embedding + LanceDB работают
- выбранный baseline retrieval не "теряет" ответ

### 2. Низкий Precision@K = 0.1356 и F1@K = 0.235
Это главный минус baseline.

Retriever часто приносит не только нужный чанк, но и много шумных фрагментов. То есть система умеет "дотянуться" до правильного ответа, но делает это не очень чисто.

Практический вывод:
- retrieval сейчас recall-oriented
- baseline хорош как MVP
- следующий шаг улучшения должен быть направлен именно на снижение шума

### 3. Неплохие MRR, MAP и NDCG
- `MRR = 0.7024`
- `MAP = 0.6829`
- `NDCG@K = 0.7528`

Это означает, что релевантные чанки часто находятся достаточно высоко в выдаче. То есть проблема baseline не в полном провале ранжирования, а скорее в том, что вокруг релевантных чанков остается лишний контекст.

### 4. Faithfulness = 0.7463
Faithfulness получился на хорошем среднем уровне для baseline.

Это значит, что генератор в большинстве случаев опирается на retrieved контекст, а не полностью галлюцинирует. Но запас для улучшения есть:
- часть ответов все еще могла включать лишние формулировки
- шумный retrieval мог ухудшать качество генерации

---

## Что baseline уже показывает
Система в текущем виде:
- работает end-to-end
- проходит по контракту оценки
- хорошо извлекает нужные фрагменты
- дает адекватный уровень faithfulness
- но retrieval пока шумный

Именно поэтому я считаю baseline успешным: он уже решает задачу как MVP, а дальнейшая работа должна идти не в сторону полной перестройки архитектуры, а в сторону локального улучшения retrieval quality.

## Следующие шаги развития

На основе текущих метрик самыми логичными следующими экспериментами выглядят:

### Гипотеза 1 - фильтрация или буст по году документа

**Проблема:** тестовый набор явно завязан на различение 2019 и 2025, а вопросы часто содержат указание на год.  
**Идея:** если в вопросе явно упоминается 2019 или 2025, можно добавлять year-aware фильтрацию или буст документов соответствующего года.  
**Ожидаемый эффект:** рост Precision@K и F1@K.

### Гипотеза 2 - reranker поверх top-k retrieval

**Проблема:** Recall очень высокий, а Precision низкий. Это классический сигнал, что retriever уже находит нужный ответ, но плохо очищает топ выдачи от шума.  
**Идея:** после первичного поиска взять top-k кандидатов и переупорядочить их отдельным reranker-модулем.  
**Ожидаемый эффект:** рост Precision@K, MAP, NDCG и потенциально Faithfulness.

### Гипотеза 3 - hybrid retrieval

**Проблема:** часть вопросов может лучше матчиться по точным словам и формулировкам, а не только по векторной близости.  
**Идея:** добавить BM25 или гибридный поиск.  
**Ожидаемый эффект:** уменьшение числа шумных чанков и рост Precision@K.
